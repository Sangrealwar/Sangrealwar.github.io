Dubbo是什么

该章节内容均来自官网

简介

一款云原生微服务开发框架。构建具备内置 RPC、流量管控、安全、可观测能力的应用，支持Kubernetes和VM部署环境。

特性

  • 高性能、可伸缩的服务发现机制，与Nacos，Zookeeper等注册中心适配；
  • 流量管控。提供路由规则，实现全链路灰度、金丝雀发布、重试等服务治理能力；
  • 灵活部署。支持虚拟机、k8s、服务网格等部署方式；
  • 多种通信协议。支持Http2、gRpc、Rpc，Rest等通信方式；
  • 可扩展。通过定制Filter、Router、Service等自定义调用，管控行为；
  • 多SDK。Java、Golang、Python等客户端；
  • 可观测。提供Metrics、Tracing、Accesslog，了解服务运行状态；
  • 认证鉴权。支持TLS链路和认证加密、基于请求鉴权；
  • 服务网格（Service Mesh）。无缝接入Istio治理体系；
  • 生态丰富。一站式微服务适配，包括注册中心、网关、限流降级、负载均衡、一致性事务、一致性消息、链路追踪等。

假设有2个服务提供者

如果是服务级别注册。注册中心数据如下。两个ip，分别有两个服务提供方，一共4条数据。

如果是应用级别注册。注册中心数据如下，两个ip，一共就2条数据。

这里会存在几个问题

  1. 主流微服务模型，如SpringCloud，K8s都是注册了应用及ip端口，没有具体的接口（rpc方法）
  2. 当某个接口类下的方法增多时，注册中心推送给各消费方的数据量会增加。一般应用数量运维可控，但是方法数量会随着开发迭代而提升，相对不可控。

为此，Dubbo有如下调整

  1. 注册中心以”应用-实例列表”格式组织数据，不保存RPC服务信息
  2. Client – Server 自行协商 RPC 方法信息

为了兼容历史接口级别注册，在配置中心会维护接口服务 -> 应用 的关系

所以使用上，需要遵循消费方指定应用，应用->接口关系存储到源数据中心，消费方和应用放可通过直接调用或调用元数据中心获取关心，从而继续接口级别调用。具体配置参考官网。

工作实践

工作中用的比较多的是Dubbo2.x版本

  • 序列化
    • 默认用的是hessian序列化，需要对象实现Serializable接口
  • 通信协议
    • 默认用rpc协议，端口20880
  • 注册中心
    • Zookeeper注册中心，或者Nacos
  • 配置中心
    • 一般默认跟着注册中心走，zookeeper或者Nacos
    • 做一些服务管理工作，比如Nacos上可以管理的服务上下线
  • Spi扩展点
    • 优雅下线扩展点，ShutdownHookCallback
    • 调用拦截扩展点，Filter
  • 服务注册级别
    • 接口服务级别。也就是类+方法级别
